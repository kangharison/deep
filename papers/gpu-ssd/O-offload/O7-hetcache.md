# [O7] HetCache: Synergising NVMe Storage and GPU Acceleration

- **학회/연도:** CIDR 2023
- **저자:**
- **분류:** GPU-NVMe 오프로드

## 핵심 요약 (1~2문장)
GPU VRAM, CPU DRAM, NVMe SSD로 구성된 이기종(heterogeneous) 메모리 계층을 통합 관리하는 캐시 시스템. 데이터베이스 쿼리 처리 등 데이터 분석 워크로드에서 GPU 가속과 NVMe 스토리지를 시너지 있게 활용하여 비용 대비 성능을 극대화.

## 읽기 전 질문
- "Synergising"이라는 표현이 의미하는 NVMe-GPU 시너지의 구체적인 메커니즘은 무엇인가?
- 캐시 교체 정책(eviction policy)이 기존 LRU/LFU와 어떻게 다른가?
- DNN 워크로드가 아닌 데이터 분석 워크로드에서 GPU+NVMe의 이점은 무엇인가?

## 주요 내용 정리

### 1. 문제 정의 (Problem)
- 현대 서버는 GPU, NVMe SSD 등 이기종 하드웨어를 장착하지만 소프트웨어가 이를 통합 활용하지 못함
- **GPU 가속 데이터 분석**: GPU VRAM이 부족하여 처리 가능한 데이터 크기가 제한됨
- **NVMe SSD의 미활용**: 대부분의 GPU 가속 시스템은 NVMe를 단순 스토리지로만 사용
- 기존 캐시 시스템은 단일 계층(CPU DRAM만) 또는 2계층에 최적화
- CPU DRAM, GPU VRAM, NVMe SSD 3계층을 모두 고려한 캐시 관리 부재
- GPUDirect Storage 등 하드웨어 기능이 존재하지만 소프트웨어 지원 미비

### 2. 제안 방법 (Approach)
- **3-Tier 캐시 계층**: NVMe SSD(대용량, 느림) → CPU DRAM(중간) → GPU VRAM(소용량, 빠름)
- **워크로드 인식 캐시 정책**: 쿼리 실행 계획을 분석하여 어떤 데이터를 어느 계층에 캐시할지 결정
- **GPUDirect Storage 활용**: NVMe → GPU VRAM 직접 전송으로 CPU 바이패스
- **비동기 프리페칭**: 다음 쿼리에 필요한 데이터를 미리 상위 계층으로 이동
- **비용 기반 배치(Cost-Based Placement)**: 각 계층의 대역폭, 용량, 비용을 고려한 최적 배치
- **핫/콜드 데이터 분류**: 접근 빈도에 따라 데이터를 적절한 계층에 자동 분배

### 3. 핵심 아키텍처/설계

```
  HetCache 3-Tier 캐시 아키텍처:
  ┌─────────────────────────────────────────┐
  │          Query Execution Engine          │

  └────────────────┬────────────────────────┘
                   │
  ┌────────────────▼────────────────────────┐
  │         HetCache Manager                 │
  │  ┌───────────────────────────────┐      │
  │  │  Workload Analyzer            │      │
  │  │  - 쿼리 실행 계획 분석        │      │
  │  │  - 데이터 접근 패턴 예측      │      │
  │  └──────────────┬────────────────┘      │
  │                 │                        │
  │  ┌──────────────▼────────────────┐      │
  │  │  Placement Policy             │      │
  │  │  - 비용 기반 최적 배치        │      │
  │  │  - 핫/콜드 분류               │      │
  │  └──────────────┬────────────────┘      │

  ┌──────┐    ┌──────────┐      ┌──────────┐

  │      │    │          │      │          │

  쿼리 요청 → VRAM hit? → 처리

              SSD에서 로드 → (DRAM 또는 GDS로 GPU) → 처리
```

### 4. 실험 결과 (Key Results)
- TPC-H, TPC-DS 등 데이터 분석 벤치마크에서 평가
- CPU DRAM만 사용하는 시스템 대비 **2~5x 쿼리 처리 성능 향상**
- GPU VRAM만 사용(데이터 크기 제한)하는 시스템과 **동일한 성능**을 더 큰 데이터셋에서 달성
- NVMe SSD 활용으로 **처리 가능한 데이터 크기 10~50배 확장**
- GPUDirect Storage로 NVMe→GPU 전송 시 CPU 경유 대비 **지연시간 30~40% 감소**
- 비용 효율성: 동일 성능 대비 **하드웨어 비용 40~60% 절감** (DRAM을 SSD로 대체)

### 5. 한계점 및 향후 연구
- 데이터 분석/쿼리 처리에 초점 → DNN 학습 워크로드에는 직접 적용 어려움
- 캐시 정책이 쿼리 실행 계획에 의존 → 비정형 워크로드에서 효과 제한적
- GPU가 1개인 환경에서 주로 평가 → 멀티 GPU 환경 확장 필요
- NVMe SSD의 쓰기 성능(write throughput) 최적화 부족
- CXL 메모리와의 통합 가능성 탐구 필요
