# [P1] GPUfs: Integrating a File System with GPUs

- **학회/연도:** ASPLOS 2014
- **저자:** Silberstein et al.
- **분류:** 선행 연구 (Pre-2023)

## 핵심 요약 (1~2문장)
GPU에서 파일 API를 제공하는 최초 연구. CPU가 I/O를 처리하되 GPU 커널에서 파일 접근 코드를 작성할 수 있게 함.

## 읽기 전 질문
- GPU 커널 내부에서 파일 시스템 API를 호출하면 CPU와의 동기화는 어떻게 처리하는가?
- GPU의 수천 개 스레드가 동시에 파일 I/O를 요청하면 병목은 어디서 발생하는가?
- 기존 CPU 기반 I/O 대비 프로그래밍 모델의 편의성은 얼마나 향상되는가?

## 주요 내용 정리

### 1. 문제 정의 (Problem)
- GPU는 강력한 병렬 연산 능력을 갖추고 있지만, 데이터 I/O는 전적으로 CPU에 의존
- GPU 프로그래머는 데이터를 미리 CPU에서 읽어 GPU 메모리로 복사해야 하며, 이는 프로그래밍 복잡도를 높임
- GPU 메모리에 들어가지 않는 대규모 데이터셋 처리 시, 수동 데이터 파티셔닝이 필요
- GPU 커널에서 직접 파일을 열고 읽는 것이 불가능하여 프로그래밍 모델이 제한적

### 2. 제안 방법 (Approach)
- **GPUfs**: GPU 커널 코드 내에서 POSIX 유사 파일 API(gopen, gread, gwrite, gclose)를 호출할 수 있는 라이브러리
- CPU 측에 GPUfs 호스트 데몬이 실제 파일 I/O를 수행하고, GPU 스레드블록은 공유 메모리 기반의 요청 큐를 통해 CPU와 통신
- GPU 메모리를 페이지 캐시로 활용하여 반복 접근 시 재사용
- **버퍼 캐시 계층**: GPU 글로벌 메모리에 페이지 캐시를 두어 CPU 페이지 캐시와 일관성 유지

```
+------------------+         +------------------+
|   GPU Kernel     |         |   CPU Host       |
|  gopen/gread/    | ------> | GPUfs Daemon     |
|  gwrite/gclose   | <------ | (실제 I/O 수행)  |
|                  |  공유   |                  |
|  [GPU Page Cache]|  메모리 | [Host Page Cache]|
+------------------+  큐     +------------------+
        |                           |
        v                           v
   GPU Global Memory           File System / Disk
```

### 3. 핵심 아키텍처/설계
- **GPU 측 API 레이어**: GPU 커널 코드에서 호출 가능한 파일 시스템 API 제공
- **요청 큐 (Request Queue)**: GPU-CPU 간 공유 메모리(pinned memory)에 I/O 요청을 큐잉
- **GPU 페이지 캐시**: GPU 글로벌 메모리의 일부를 페이지 캐시로 사용하여 반복 접근 최적화
- **일관성 프로토콜**: GPU 캐시와 CPU 캐시 간 약한 일관성 모델 사용
- **Warp 수준 협력**: 같은 warp 내 스레드들이 협력하여 coalesced I/O 요청을 생성

### 4. 실험 결과 (Key Results)
- GPU 자체 파일 접근으로 **프로그래밍 코드량 대폭 감소** (수동 데이터 관리 코드 제거)
- External sort 벤치마크에서 CPU 대비 **최대 4x 성능 향상**
- GPU 페이지 캐시 적중 시, 호스트 메모리 접근 대비 낮은 레이턴시
- GPU 메모리보다 큰 데이터셋도 자동 페이지 관리를 통해 처리 가능
- 단, I/O 집약적 워크로드에서는 CPU-GPU 간 통신 오버헤드로 성능 제한

### 5. 한계점 및 향후 연구
- **CPU 경유 I/O**: 모든 I/O가 CPU 데몬을 거쳐야 하므로 CPU가 병목 가능
- **GPU-SSD 직접 통신 불가**: P2P DMA 미지원으로 데이터가 항상 CPU 메모리를 경유
- **제한된 일관성 모델**: 강한 일관성 보장이 어려움
- **GPU 페이지 캐시 크기 제한**: GPU 메모리가 연산과 캐시를 공유하므로 트레이드오프 존재
- 향후: GPU-SSD 직접 접근 (NVMMU [P2], BaM [I1]으로 발전)

## 다른 논문과의 관계
- 선행 연구: GPGPU 프로그래밍 모델 (CUDA, OpenCL)
- 후속 연구: BaM [I1], GoFS [I4], GeminiFS [I5] - GPUfs의 CPU 경유 한계를 극복
- 비교 대상: 전통적 CPU-GPU 데이터 전송 방식 (cudaMemcpy)
