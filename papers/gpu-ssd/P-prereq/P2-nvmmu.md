# [P2] NVMMU: A Non-volatile Memory Management Unit for Heterogeneous GPU-SSD Architectures

- **학회/연도:** PACT 2015
- **저자:** Zhang et al.
- **분류:** 선행 연구 (Pre-2023)

## 핵심 요약 (1~2문장)
GPU-SSD 간 직접 접근을 위한 하드웨어 메모리 관리 유닛 제안. GPU에서 스토리지에 직접 접근하는 개념의 초기 하드웨어 수준 연구.

## 읽기 전 질문
- GPU 주소 공간에서 NVM/SSD 주소를 어떻게 매핑하는가?
- 하드웨어 수준의 주소 변환이 소프트웨어 방식 대비 얼마나 빠른가?
- GPU 워프 단위의 메모리 접근 패턴이 SSD 접근 패턴과 어떻게 조화를 이루는가?

## 주요 내용 정리

### 1. 문제 정의 (Problem)
- GPU가 SSD의 데이터에 접근하려면 반드시 CPU를 경유해야 함 (GPUfs의 한계)
- CPU 경유 방식은 데이터 복사 오버헤드와 CPU 병목을 유발
- GPU와 NVM/SSD 간 직접 데이터 경로가 없으며, 주소 변환 메커니즘도 부재
- GPU의 가상 메모리 시스템이 스토리지 디바이스를 인식하지 못함

### 2. 제안 방법 (Approach)
- **NVMMU (Non-volatile Memory Management Unit)**: GPU 내에 하드웨어 주소 변환 유닛을 추가하여 GPU 가상 주소를 NVM/SSD 물리 주소로 직접 매핑
- GPU 프로그램이 NVM 영역의 가상 주소에 접근하면, NVMMU가 이를 감지하여 SSD로 직접 I/O 요청 생성
- 페이지 폴트 기반 메커니즘: GPU 스레드가 NVM 매핑 주소에 접근 시 폴트 발생 → NVMMU가 SSD에서 데이터 fetch

```
+------------------+
|   GPU Core       |
|   (CUDA Thread)  |
|   load/store     |
+--------+---------+
         |  가상 주소
+--------v---------+
|     NVMMU        |  ← 새로 추가된 하드웨어
|  (주소 변환 +    |
|   DMA 엔진)      |
+--------+---------+
         |  물리 주소 / DMA 요청
+--------v---------+
|   NVM / SSD      |
|  (PCIe 연결)     |
+------------------+
```

### 3. 핵심 아키텍처/설계
- **주소 변환 테이블**: GPU 가상 주소 → NVM 물리 주소 매핑 테이블을 NVMMU 내부에 유지
- **페이지 폴트 처리**: GPU 스레드가 NVM 주소 접근 시 NVMMU가 폴트를 처리하고 DMA로 데이터 로드
- **GPU 캐시 통합**: 로드된 NVM 데이터를 GPU L2 캐시 또는 글로벌 메모리에 캐싱
- **쓰기 경로**: GPU에서 수정된 데이터를 NVM으로 write-back하는 메커니즘
- **멀티 스레드 요청 병합**: 다수의 GPU 스레드가 인접 NVM 주소를 요청할 때 요청을 병합하여 SSD I/O 효율성 향상

### 4. 실험 결과 (Key Results)
- 시뮬레이션 기반 평가 (GPGPU-Sim + SSD 시뮬레이터)
- CPU 경유 방식 대비 **약 2x 데이터 전송 속도 향상**
- GPU-SSD 직접 경로로 CPU 사용률 대폭 감소
- 페이지 사이즈에 따른 성능 민감도 분석: 큰 페이지(4KB~64KB)가 SSD 접근에 유리
- 그래프 탐색, 데이터베이스 스캔 등 데이터 집약적 워크로드에서 효과 입증

### 5. 한계점 및 향후 연구
- **시뮬레이션 기반 평가**: 실제 하드웨어 구현이 아닌 시뮬레이터 기반이므로 실제 성능과 차이 가능
- **하드웨어 변경 필요**: GPU 아키텍처 자체를 수정해야 하므로 상용화 장벽이 높음
- **NVM 기술 의존**: 당시 NVM 기술(PCM 등)의 성숙도가 낮았음
- **일관성 문제**: 멀티 GPU 환경에서의 일관성 프로토콜은 미해결
- 향후: FlashGPU [P3]에서 플래시를 GPU에 더 가깝게 배치하는 방향으로 발전

## 다른 논문과의 관계
- 선행 연구: GPUfs [P1] (소프트웨어 기반 GPU 파일 접근)
- 후속 연구: FlashGPU [P3] (플래시를 GPU 패키지 내부에 배치)
- 비교 대상: GPUfs [P1]의 소프트웨어 방식 vs NVMMU의 하드웨어 방식
