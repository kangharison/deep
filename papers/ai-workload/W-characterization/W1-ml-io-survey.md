# [W1] I/O in Machine Learning Applications on HPC Systems: A 360-degree Survey ★필독

- **학회/연도:** ACM Computing Surveys 2024 (2025 출판)
- **저자:** Bin Dong et al. (Lawrence Berkeley National Laboratory)
- **분류:** 워크로드 특성화 & 서베이
- **DOI:** 10.1145/3722215

## 핵심 요약 (1~2문장)
2019~2024년 6년간의 문헌을 분석하여, ML 애플리케이션의 I/O를 데이터 준비 → 트레이닝 → 추론의 전체 라이프사이클에 걸쳐 360도 서베이한 논문. HPC 시스템에서 ML 워크로드의 I/O 패턴이 전통적 과학 시뮬레이션과 근본적으로 다름을 체계적으로 보여준다.

## 읽기 전 질문
- ML 워크로드의 I/O가 전통 HPC I/O와 구체적으로 어떻게 다른가?
- 데이터 준비, 트레이닝, 추론 각 단계에서 스토리지 병목의 성격이 어떻게 달라지는가?
- 기존 병렬 파일시스템(Lustre, GPFS 등)이 ML I/O에 왜 비효율적인가?

## 주요 내용 정리

### 1. 문제 정의 (Problem)
- ML 워크로드가 HPC 센터의 주요 워크로드로 급부상했으나, 스토리지 시스템은 전통 시뮬레이션(대용량 순차 I/O) 기준으로 설계됨
- ML의 I/O 패턴은 **수많은 작은 파일에 대한 랜덤 읽기**가 지배적 — 기존 병렬 FS의 최적화가 맞지 않음
- 라이프사이클 단계별로 I/O 특성이 크게 다르지만, 이를 통합적으로 분석한 서베이가 없었음

### 2. ML I/O 라이프사이클별 특성

```
┌─────────────────────────────────────────────────────────────────┐
│                    ML I/O 라이프사이클                           │
│                                                                 │
│  ┌──────────────┐   ┌──────────────┐   ┌──────────────┐       │
│  │  데이터 준비   │──→│   트레이닝    │──→│    추론       │       │
│  │  (Offline)    │   │  (Iterative) │   │  (Online)    │       │
│  └──────┬───────┘   └──────┬───────┘   └──────┬───────┘       │
│         │                  │                   │               │
│    대용량 순차 R/W     에폭 반복 읽기       모델 로딩 +         │
│    ETL 파이프라인     셔플 → 랜덤화        소량 입력 읽기       │
│    데이터 포맷 변환    체크포인트 쓰기      배치 단위 처리       │
│    증강(augmentation) 로그/메트릭 쓰기     KV Cache 관리       │
│                                                                 │
│  I/O 크기: 대 (MB~GB)  중 (KB~MB)         소 (KB) + 대 (GB)   │
│  접근패턴: 순차          랜덤+순차          랜덤               │
│  빈도:    1회(배치)      반복(에폭)         연속(스트리밍)      │
└─────────────────────────────────────────────────────────────────┘
```

### 3. 핵심 발견사항

**ML I/O vs 전통 HPC I/O 차이점:**
- **파일 크기**: ML은 수백만 개의 작은 파일(이미지, 오디오 등)을 사용. 전통 HPC는 소수의 대용량 파일
- **접근 패턴**: ML은 에폭마다 전체 데이터셋을 셔플하며 랜덤 순서로 읽음. 전통 HPC는 순차 접근 지배적
- **메타데이터 부하**: 수백만 파일의 open/stat/close가 메타데이터 서버에 과부하
- **읽기 지배적**: 트레이닝 중 읽기:쓰기 비율이 약 100:1 (체크포인트 외에는 거의 읽기만)

**주요 I/O 최적화 기법 분류:**
- **데이터 포맷**: TFRecord, WebDataset, LMDB 등으로 작은 파일을 큰 sequential 파일로 묶기
- **캐싱**: 로컬 SSD, burst buffer, in-memory 캐시 활용
- **프리페치**: 다음 배치를 GPU 연산 중 미리 읽기 (파이프라이닝)
- **데이터 로더**: PyTorch DataLoader, DALI 등의 비동기 멀티프로세스 로딩

### 4. 서베이 범위 및 방법론
- 6년간(2019~2024) 주요 학회/저널 논문 체계적 분석
- 데이터 준비, 트레이닝, 추론 각 단계별 I/O 최적화 기법 분류
- 현재 ML 프레임워크(PyTorch, TensorFlow, JAX)의 I/O 구현 분석

### 5. 한계점 및 향후 연구
- LLM 특화 I/O 패턴(KV Cache, 초대규모 체크포인트)은 아직 충분히 다루지 못함
- GPU-initiated I/O, CSD 기반 접근 등 최신 하드웨어 혁신은 범위 외
- 멀티모달(이미지+텍스트+비디오) 워크로드의 I/O 특성은 추가 연구 필요

## 다른 논문과의 관계
- 이 서베이가 다루는 범위 안에 있는 논문: [W2] LLM 데이터센터 특성화, [T1] DataStates-LLM
- 이 서베이 이후 트렌드를 다루는 논문: [S1] Mooncake, [S3] NVMe KV 오프로드 (KV Cache 관련)
- gpu-ssd 연결: 이 서베이의 "작은 랜덤 읽기" 특성이 BaM [I1], GIDS [I2]의 동기

## 발표 자료 활용 포인트
- ML I/O 라이프사이클 다이어그램을 발표 도입부에 활용
- "ML I/O는 전통 HPC I/O와 다르다"는 메시지의 근거 논문
- 핵심 수치: ML 워크로드의 읽기:쓰기 비율 ~100:1

## 메모
- AI I/O 연구의 출발점으로 가장 적합한 서베이. 이 논문을 먼저 읽고 관심 분야를 좁혀나가는 것을 추천
- gpu-ssd 논문들이 다루는 "GPU에서 직접 스토리지 접근" 기법은 이 서베이에서는 아직 주류가 아님 — 향후 서베이에서 다뤄질 영역
