# [W1] I/O in Machine Learning Applications on HPC Systems: A 360-degree Survey ★필독

- **학회/연도:** ACM Computing Surveys, Vol. 1, No. 1 (January 2025)
- **저자:** Noah Lewis (Ohio State Univ. & Louisiana State Univ.), Jean Luca Bez (Lawrence Berkeley National Lab), Suren Byna (Ohio State Univ.)
- **분류:** 워크로드 특성화 & 서베이
- **DOI:** 10.1145/3722215
- **서베이 규모:** 39편 논문, 2019~2024년, ACM Digital Library + IEEE Xplore

## 핵심 요약 (1~2문장)
ML 애플리케이션의 I/O를 데이터 준비 → 트레이닝 → 추론의 전체 라이프사이클에 걸쳐 360도 서베이한 논문. 39편의 문헌을 체계적으로 분석하여 ML 워크로드의 I/O 패턴이 전통 HPC 시뮬레이션과 근본적으로 다름을 보이고, 프레임워크별(PyTorch 30편, TensorFlow 23편) I/O 특성과 최적화 기법을 종합한다.

## 읽기 전 질문
- ML 워크로드의 I/O가 전통 HPC I/O와 구체적으로 어떻게 다른가?
- 데이터 준비, 트레이닝, 추론 각 단계에서 스토리지 병목의 성격이 어떻게 달라지는가?
- 기존 병렬 파일시스템(Lustre, GPFS 등)이 ML I/O에 왜 비효율적인가?

### 2. 단계별 I/O 패턴 상세

#### Phase 2: 오프라인 데이터 준비
- 순차 I/O가 지배적 (TFRecord, JSON 등 랜덤 접근 불가 포맷)
- Sharding 전략: 대규모 데이터셋을 작은 파일로 분할 → 병렬 처리
- 각 프로세스가 할당된 샤드를 순차적으로 읽음
- 메타데이터 오버헤드가 많은 파일 관리 시 심각

#### Phase 3: 트레이닝 (DLIO 벤치마크 실측 결과)

**Unet3D 워크로드 (3D 의료 영상):**
```
설정: 4노드, 노드당 4프로세스, CPU 코어 4개/프로세스
데이터: 168개 샘플, 파일당 1개 샘플, ~146 MiB/NPZ 파일
I/O 패턴:
  - 파일↔샘플 1:1 대응
  - 같은 rank(rank 14)가 같은 파일을 에폭마다 반복 접근
  - 캐싱 미사용 → 매번 PFS까지 도달
  - 10 에폭 동안 에폭 5,7,9에서 체크포인트 (3회)
```

**BERT 워크로드 (LLM):**
```
설정: 8노드, 노드당 4프로세스 (총 32프로세스), CPU 코어 2개/프로세스
데이터: 10개 TFRecord 파일, 파일당 31,353 샘플, 샘플 크기 2,500 bytes
       총 313,530 샘플
I/O 패턴:
  - 배치 크기 48: ⌈313,530 ÷ (32×48×10)⌉ = 에폭당 21회 I/O
  - 배치 크기 96: 에폭당 11회 I/O (예측 가능한 선형 스케일링)
  - 32개 프로세스가 각 배치 읽기 후 동기화
  - 체크포인트: 250 스텝마다 → 배치48 기준 78개 체크포인트
```

**체크포인트 I/O:**
- 모델 파라미터가 쓰기 크기 결정: GPT-3 = 175B params × 4bytes = ~700GB
- 기본 정밀도 32-bit → 파라미터당 4 bytes
- 빈번한 체크포인트 시 쓰기가 I/O를 지배

#### Phase 4: 추론
- **오프라인 추론**: PFS에서 대용량 순차 읽기, 배치 처리, 높은 처리량
- **스트리밍 추론**: 클라이언트 디바이스가 단일 샘플 쿼리를 병렬 전송, 지연시간+대역폭 제약 심각, 로드 밸런싱 필수

### 3. ML I/O vs 전통 HPC I/O 핵심 차이

| 특성 | 전통 HPC (시뮬레이션) | ML 워크로드 |
|------|---------------------|------------|
| 파일 수 | 소수의 대용량 파일 | 수백만 개의 작은 파일 |
| 접근 패턴 | 순차 접근 지배적 | SGD 셔플링 → 랜덤 접근 |
| 메타데이터 부하 | 낮음 | 높음 (open/stat/close 수백만 건) |
| 읽기/쓰기 비율 | 혼합 | 읽기 지배적 (~100:1, 체크포인트 외) |
| 주요 언어 | C/C++/Fortran + MPI | Python + PyTorch/TF |
| I/O 스택 | MPI-IO → HDF5 → PFS | DataLoader → POSIX → PFS |

### 4. 데이터 포맷 비교 (Table 2)

| 포맷 | 병렬 I/O | 청킹 | 부분 압축해제 | 스키마 | 바이너리 |
|------|:-------:|:----:|:----------:|:-----:|:------:|
| **HDF5** | ✓ | ✓ | ✓ | ✓ | ✓ |
| **Apache Parquet** | ✓ | ✓ | ✓ | ✓ | ✓ |
| **TFRecord** | ✗ | ✗ | ✓ | ✓ | ✓ |
| **RecordIO** | ✗ | ✓ | ✓ | ✗ | ✓ |
| **NPZ** | ✗ | ✗ | ✗ | ✗ | ✓ |
| **이미지/비디오/텍스트** | ✗ | ✗ | ✗ | ✗ | 혼합 |

- **고성능 포맷 (HDF5, Parquet)**: 청킹으로 Lustre 스트라이핑과 정렬 가능, 병렬 I/O 지원
- **직렬화 포맷 (TFRecord, NPZ)**: 순차 접근만 가능, 스트리밍에 효율적이나 랜덤 접근 불가
- **핵심 기능 — 부분 압축해제**: 전체 파일 로딩 없이 부분 데이터만 읽기 가능 → ML의 서브셋 접근에 중요

### 6. 프레임워크별 I/O 메커니즘

#### PyTorch DataLoader
- `torch.utils.data.DataLoader`: Python iterable 기반
- **Map-style**: 인덱스 접근, 노드-로컬 메모리에 fit하는 경우
- **Iterable-style**: 스트리밍, 랜덤 접근 비싼 시나리오
- `num_workers`: 워커 프로세스 수 (비동기 프리페치)
- `pin_memory=True`: 페이지 고정 → GPU DMA 가속
- 체크포인트: `torch.save()`/`torch.load()` (pickle 기반), 분산 체크포인트 API (2023.10, v2.12.1)

#### TensorFlow tf.data
- `tf.data.Dataset`: 컴포저블 파이프라인
- `prefetch()`: 다음 배치 미리 로딩
- `cache()`: 인메모리 캐싱
- `distributed_cache()`: 멀티노드 분산 캐싱
- `interleave()`: 여러 파일 동시 읽기
- 체크포인트: 워커별 독립 저장 (3파일: index, metadata, data), 단일 워커 방식은 straggler 문제

#### NVIDIA DALI
- GPU에서 직접 배치 전처리 수행
- 이미지/비디오 데이터셋에 최적화
- DLIO 벤치마크와 통합

### 7. 스토리지 시스템 분석

**Perlmutter 슈퍼컴퓨터 (NERSC) 테스트 환경:**
```
파일시스템: All-flash Lustre PFS
대역폭:    >5 TB/sec (집계)
IOPS:      4백만 ops (4KiB 랜덤)
MDS:       16개 메타데이터 서버
OST:       298개 오브젝트 스토리지 타겟
스트라이프: 1 MiB (워크로드별 조정 가능)
```

**ML 워크로드의 PFS 문제점:**
- 수백만 작은 파일의 메타데이터 조회가 MDS에 과부하
- SGD 셔플링이 순차 접근에 최적화된 PFS와 불일치
- Burst Buffer(노드-로컬 SSD)가 활용되지 않는 경우 많음 → ML 개발자가 HPC 인프라에 익숙하지 않음

### 8. I/O 최적화 기법 종합

| 기법 | 설명 | 프레임워크 지원 |
|------|------|---------------|
| 프리페치 | GPU 연산 중 다음 배치 미리 읽기 | PyTorch(`num_workers`), TF(`prefetch()`) |
| 캐싱 | 인메모리/로컬SSD 캐시 | TF(`cache()`), PyTorch(커스텀) |
| 비동기 체크포인트 | 체크포인트를 백그라운드로 | PyTorch 미지원(빌트인), 커스텀 필요 |
| 분산 체크포인트 | 다수 rank가 병렬 쓰기 | PyTorch v2.12.1+(2023.10), TF 미지원 |
| 데이터 포맷 변환 | 작은 파일→큰 순차 파일 | TFRecord, WebDataset, LMDB |
| 모델 압축 | 양자화/프루닝으로 체크포인트 축소 | 프레임워크 독립 |

### 9. I/O 프로파일링 도구

| 도구 | 대상 | 한계 |
|------|------|------|
| **Darshan** | POSIX/STDIO 레벨 I/O 추적 | MPI 애플리케이션 중심 |
| **tf-Darshan** | TF 런타임 I/O 분석 | TF 전용, 10~20% 오버헤드 |
| **TF Profiler** | 호스트+GPU 트레이싱 | TF 레벨만, POSIX 미포착 |
| **DLIO** | ML I/O 벤치마크 시뮬레이션 | 2D 이미지 가정, 포맷 제한 |

### 10. 미해결 과제 및 향후 연구 방향

1. **프로파일링 도구 한계**: 스레드 레벨 I/O 식별 불가, MPI 중심 도구로 GLOO/Spark/Dask 비지원
2. **프레임워크 호환성**: TFRecord↔PyTorch 간 포맷 호환 문제, 벤치마크 시 포맷 변환 필요
3. **스토리지-워크로드 불일치**: PFS가 대용량 순차에 최적화 ↔ ML은 작은 랜덤 읽기 → 전용 스토리지 설계 필요
4. **클라우드 통합**: BigQuery/Redshift 등 원격 데이터 웨어하우스 접근 지연시간
5. **추론 I/O 특성화 부족**: 서베이 39편 중 추론 관련은 2편뿐 → 온라인/오프라인/에지 추론 I/O 분류 필요
6. **멀티모달 데이터**: 이미지+텍스트+비디오 혼합 워크로드의 I/O 특성 연구 미흡

## 메모
- 서베이된 39편 중 PyTorch 30편, TF 23편 → PyTorch가 ML I/O 연구의 주류 프레임워크
- 트레이닝 32편 vs 추론 2편 → 추론 I/O는 아직 미개척 영역 (LLM 추론의 KV Cache 문제가 이 gap)
- Perlmutter의 all-flash Lustre가 5TB/s인데도 ML I/O에는 부족 → 문제가 대역폭이 아니라 접근 패턴에 있음
