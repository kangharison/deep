# [W2] Characterization of Large Language Model Development in the Datacenter

- **학회/연도:** NSDI 2024
- **저자:** Qinghao Hu, Zhisheng Ye (NTU, Shanghai AI Lab), Zerui Wang (SJTU), Guoteng Wang, Meng Zhang, Qiaoling Chen (Shanghai AI Lab), Peng Sun (SenseTime), Dahua Lin (CUHK), Xiaolin Wang, Yingwei Luo (Peking Univ.), Yonggang Wen, Tianwei Zhang (NTU)
- **분류:** 워크로드 특성화
- **트레이스 공개:** https://github.com/InternLM/AcmeTrace

## 핵심 요약 (1~2문장)
GPU 데이터센터 "Acme"에서 6개월간(2023.3~8) 수집한 109만 건 LLM 개발 워크로드 트레이스의 최초 대규모 특성화 연구. 기존 DL 트레이스(Philly, Helios, PAI)와 LLM 워크로드의 근본적 차이를 실측으로 밝히고, 장애 진단과 평가 최적화 솔루션을 제안한다.

## 읽기 전 질문
- LLM 트레이닝 작업의 실제 GPU 사용 패턴은 기존 DL과 어떻게 다른가?
- 프로덕션 LLM 트레이닝에서 가장 빈번한 장애 유형과 그 I/O 영향은?
- LLM 개발 라이프사이클(프리트레이닝, SFT, RLHF, 평가)별 리소스 사용 패턴 차이는?

### 4. GPU 활용 패턴 (Figure 7)

**SM(Streaming Multiprocessor) 활용률:**
- Kalos median: ~40% (PAI 20%의 2배)
- 양극화 패턴: 0% 또는 100%에 집중 → transformer의 균일한 연산 특성 반영

**GPU 메모리 사용:**
- 50%의 GPU가 >75% GPU 메모리(60GB+) 사용
- PAI 대비 훨씬 높은 메모리 압박 → LLM의 대규모 파라미터 + KV Cache

**CPU 활용률:** <50% — 심각하게 미활용
**InfiniBand:** >60% 시간 유휴 — 네트워크도 미활용
**호스트 메모리:** <50% 사용 → 체크포인트 버퍼링에 활용 가능 (T1~T4의 근거)

**핵심 수치:**
- 네트워크 에러의 재시작/장애 비율이 **107.74%** — 재시작 시간이 실행 시간보다 길 수 있음

#### 프레임워크 에러 (8.28% of GPU 시간):
- OOM: 14건, 37.89% 재시작 오버헤드
- Runtime/Assertion Error: 170건

#### 스크립트 에러 (~3% of GPU 시간):
- File Not Found, Type Error 등: 1,717건 (가장 빈번하지만 빠른 종료)

### 6. I/O 패턴: 체크포인트와 데이터 로딩

**비동기 체크포인트 개선:**
- 체크포인트 시간 **3.6~58.7배** 가속 달성
- 전략: GPU 상태를 Host Memory에 저장 → 비동기 스레드가 원격 NVMe에 저장
- 이전 동기 체크포인트: 트레이닝 **최대 43% 감속** 유발

**호스트 메모리 활용:**
- CPU 메모리 사용률 <50% → 체크포인트 버퍼링에 충분한 여유
- Kalos 노드당 2TB → 대부분 미활용 → [T4] LMStor의 dual-replica 아이디어 근거

**평가 작업의 모델 로딩 I/O:**
- 단일 GPU 트라이얼: 25Gbps NIC 대역폭 한계 (스토리지 병목)
- 1→8개 동시 트라이얼: 대역폭 심각하게 저하
- 해결: 선행 작업이 모델을 로컬 공유 메모리에 프리로딩 → PCIe로 고속 접근

**평가 작업 시간 분석 (HumanEval, 7B LLM):**
- GPU 추론: ~50%
- **모델 로딩: 29.5%** (>1분, 7B 모델)
- 프로그램 정확성 테스트: 19.0% (GPU 유휴)
- GPT-4 API 호출: 최대 30분 (GPU 유휴)

### 7. 제안 솔루션

#### A. Fault-Tolerant Pretraining

**3개 모듈:**
1. **비동기 체크포인트**: 미활용 Host Memory를 버퍼로 → 별도 스레드가 원격 저장소에 저장 (3.6~58.7× 개선)
2. **장애 진단 (Rule + LLM 기반)**:
   - 실시간 로그 압축: 필터 규칙으로 정상 출력 제거
   - LLM Log Agent: 고정 패턴 식별, 필터 규칙 동적 갱신
   - Failure Agent (GPT-4): 에러 유형 분류, user vs infrastructure 판별, 완화 제안
3. **빠른 복구**: 2라운드 NCCL 테스트로 장애 노드 식별 → 이전 정상 체크포인트에서 재시작

**효과: 수동 개입 ~90% 감소**

#### B. Decoupled Scheduling for Evaluation

1. **분리형 모델 로딩**: 선행 작업이 로컬 공유 메모리에 모델 프리로딩 → 스토리지 NIC 병목 해소
2. **분리형 메트릭 연산**: GPU가 출력 저장 후 즉시 해제, CPU가 별도로 메트릭 계산
3. **Prior-based Elastic Scheduling**: 과거 실행 시간 기반 배치 최적화

**효과: makespan 1.3× (단일노드) ~ 1.8× (4노드) 개선**

## 다른 논문과의 관계
- 체크포인트 병목 실측 → [T1] DataStates-LLM, [T2] PCcheck의 동기
- Host Memory 미활용 발견 → [T4] LMStor의 dual-replica 캐시 근거
- 40% 장애율 → [W3] 85,000 체크포인트의 체크포인트 빈도 연구 동기
- 모델 로딩 병목 → [S1] Mooncake의 KV Cache 재사용 동기
- gpu-ssd 연결: 체크포인트 I/O + 모델 로딩에 GDS P2P DMA 활용 가능

## 발표 자료 활용 포인트
- "LLM 작업 median 2분" vs "프리트레이닝이 GPU 시간 94% 차지" — 강렬한 대비
- 장애 통계표 (Table 3): 네트워크 에러의 TR/TF=107.74%
- GPU 활용 양극화 그래프 (Figure 7)
- LLM 개발 라이프사이클 → 각 단계별 I/O 특성

## 메모
- 핵심 교훈: LLM 트레이닝의 실제 비용 = 순수 연산 + 장애 복구 + 체크포인트 + 리소스 대기 → 후자가 더 클 수 있음
- CPU/메모리/IB가 모두 미활용 → Smart-Infinity [T3]나 Near-Storage Processing이 이 유휴 자원을 활용하는 방향
- Acme 트레이스가 공개(AcmeTrace)되어 있어 후속 연구에 직접 활용 가능
