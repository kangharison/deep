# [R2] Athena: A Plug-and-Play Advisor for Retrieval-Augmented Generation

- **학회/연도:** IISWC 2025
- **저자:** Liang et al. (Duke University)
- **분류:** RAG 벤치마크

## 핵심 요약 (1~2문장)
RAG 파이프라인의 각 컴포넌트(임베딩, 검색, 청킹, 생성 등)를 교체 가능한(plug-and-play) 형태로 구성하고, 다양한 조합의 성능을 체계적으로 벤치마크하는 프레임워크. RAG 파이프라인의 성능 병목이 어디에 있는지를 정량적으로 분석한다.

## 읽기 전 질문
- RAG 파이프라인의 어떤 단계가 성능 병목인가?
- 임베딩 모델, 벡터 DB, 청킹 전략 등의 선택이 전체 성능에 미치는 영향은?
- 스토리지 계층(DRAM vs NVMe)이 RAG 성능에 미치는 영향은?

## 주요 내용 정리

### 1. 문제 정의 (Problem)
- RAG 파이프라인은 다양한 컴포넌트의 조합으로 구성됨
- 각 컴포넌트 선택(임베딩 모델, 벡터 DB 엔진, 청킹 크기 등)이 성능에 복합적으로 영향
- 체계적인 벤치마크 없이 최적 구성을 찾기 어려움

### 2. 핵심 분석 결과

**RAG 파이프라인 구성 요소:**
```
문서 수집 → 청킹 → 임베딩 → 벡터 DB 저장 → (오프라인)
                                    │
사용자 쿼리 → 임베딩 → ANN 검색 → 문서 검색 → 증강 → LLM 생성 (온라인)
```

**성능 영향 요인:**
- 임베딩 모델 선택: 정확도와 속도의 트레이드오프
- 벡터 DB 엔진 (FAISS, Milvus 등): 인덱스 유형에 따른 검색 속도 차이
- 청킹 크기: 작으면 정확하지만 검색 부하 증가
- 스토리지 계층: 벡터 인덱스가 NVMe에 있으면 검색 지연 증가

### 3. 한계점 및 향후 연구
- 벤치마크 워크로드의 대표성
- 멀티모달 RAG에 대한 확장
- 실시간 인덱스 업데이트 시나리오

## 다른 논문과의 관계
- 보완: [R1] RAGX (Athena가 발견한 병목을 RAGX가 하드웨어로 해결)
- 맥락: [W1] ML I/O 서베이 (RAG I/O는 서베이에서 충분히 다루지 못한 새로운 영역)

## 발표 자료 활용 포인트
- RAG 파이프라인 전체 구조도
- 컴포넌트별 성능 영향 분석 결과

## 메모
- RAG의 I/O 특성 이해를 위한 기초 자료 — RAGX [R1]을 읽기 전에 이 논문으로 RAG 파이프라인 이해
- 벡터 DB의 NVMe 계층화 문제는 IMPRESS [S5]의 KV Cache 계층화와 유사한 구조
