# [G1] BeaconGNN: Large-Scale GNN Acceleration with Out-of-Order Streaming In-Storage Computing ★필독

- **학회/연도:** HPCA 2024 (**Best Paper Honorable Mention**)
- **저자:** Yuyue Wang, Xiurui Pan, Yuda An (UCLA), Jie Zhang (Peking Univ.), Glenn Reinman (UCLA)
- **분류:** GNN 스토리지 가속

## 핵심 요약 (1~2문장)
SSD 내부에서 GNN의 이웃 샘플링, 특성 테이블 조회, GNN 연산을 모두 수행하는 In-Storage Computing 시스템. 기존 ISC의 3가지 근본적 한계(순차 샘플링, 접근 단위 불일치, 펌웨어 병목)를 Out-of-Order 샘플링과 컨트롤러/채널/다이 3계층 Near-Data Processing으로 해결하여 HPCA 2024 Best Paper Honorable Mention을 수상했다.

## 읽기 전 질문
- GNN 워크로드의 I/O가 왜 기존 스토리지에서 극도로 비효율적인가?
- "Out-of-Order" 이웃 샘플링이 플래시 내부 병렬성을 어떻게 활용하는가?
- SSD 내부의 3계층(컨트롤러, 채널, 다이) Near-Data Processing이란?

## 주요 내용 정리

### 1. 문제 정의: GNN의 극소 랜덤 I/O

**GNN 워크로드의 I/O 특성:**
- 이웃 샘플링(Neighbor Sampling): 그래프 구조를 따라 이웃 노드 탐색
- **I/O 크기: ~64B** (노드 ID + 이웃 리스트 포인터)
- NVMe 최소 접근 단위: **4KB 페이지**
- → **읽기 증폭: 최대 64× (64B 유효 / 4KB 전체 페이지)**
- 특성 테이블 조회: 노드 특성 벡터(수십~수백 B)도 4KB 페이지 중 일부만 유효

**기존 In-Storage Computing의 3가지 한계:**
1. **순차 샘플링**: GNN의 이웃 탐색 순서를 그대로 따름 → 플래시 내부 병렬성(채널/다이) 활용 불가
2. **접근 단위 불일치**: 64B 데이터를 위해 4KB 페이지 전체 읽기 → 채널 대역폭 낭비
3. **펌웨어 병목**: FTL(Flash Translation Layer) 처리가 임베디드 코어 성능에 제약 → I/O 처리량 한계

### 3. Out-of-Order 이웃 샘플링

**기존 (In-Order):**
```
노드 A → 이웃 B → 이웃 C → 이웃 D → ...
(그래프 구조를 순서대로 따라감)
→ B, C, D가 같은 채널/다이에 있으면 직렬 접근 (병렬성 낭비)
```

**BeaconGNN (Out-of-Order):**
```
대기 큐의 모든 이웃 요청을 수집
→ 물리적 위치(채널/다이)별로 재정렬
→ 다른 채널의 노드를 동시에 접근 (병렬성 극대화)
→ 결과를 원래 순서로 재조립
```

**새로운 그래프 포맷:**
- 그래프 구조를 플래시의 물리적 레이아웃에 맞게 재배치
- 인접 노드를 가능한 한 다른 채널에 분산 → 접근 시 자연스러운 병렬성
- 전처리 단계에서 한 번 수행 (오프라인)

### 4. 다이 레벨 최적화: 부분 읽기

**기존:** 4KB 페이지 전체를 채널 버스로 전송 → 64B만 사용하고 나머지 폐기
**BeaconGNN:** 다이 레벨에서 유효 바이트만 추출하여 채널 버스로 전송
→ **채널 대역폭 활용 효율 대폭 향상**
→ 같은 채널 대역폭으로 더 많은 유효 데이터 전송 가능

### 5. 호스트의 역할

- 호스트 애플리케이션: GNN 태스크 전체 실행 제어
- **모든 데이터 집약적 연산은 SSD로 오프로드:**
  - 이웃 샘플링 (그래프 구조 탐색)
  - 특성 테이블 조회 (노드 특성 벡터)
  - GNN 연산 (이웃 집계)
- 호스트↔SSD 간 전송: 최소화 (최종 결과만)

### 6. 실험 결과
- 기존 호스트 기반 GNN 시스템 대비 대폭적인 성능 향상
- 읽기 증폭 대폭 절감
- 플래시 내부 병렬성 활용률 크게 개선
- **HPCA 2024 Best Paper Honorable Mention 수상**

### 7. 한계점 및 향후 연구
- SSD 펌웨어/하드웨어 수정 필요 → 상용 SSD에 바로 적용 불가
- 그래프 포맷 변환(전처리) 오버헤드 → 정적 그래프에 적합
- 동적 그래프(노드/엣지 실시간 추가/삭제)에 대한 대응 미비
- GNN 모델 종류(GCN, GraphSAGE, GAT 등)별 범용성 검증 필요

## 다른 논문과의 관계

**GNN I/O 가속 논문 계보:**
| 접근 | 논문 | 핵심 아이디어 |
|------|------|-------------|
| GPU→SSD 직접 접근 | BaM [I1] | GPU가 NVMe 큐 직접 조작 |
| GPU→SSD GNN 특화 | GIDS [I2] | BaM + GNN 캐싱 최적화 |
| **SSD 내부 연산** | **BeaconGNN [G1]** | OoO 샘플링 + 3계층 NDP |
| SSD 내부 연산 | FlashGNN [G2] | PCIe 우회 + 데이터 재사용 |

- BaM/GIDS: "GPU에서 SSD로의 I/O 경로 최적화"
- BeaconGNN/FlashGNN: "I/O 자체를 SSD 내부에서 해결"
- → 두 접근은 상호 보완적: GPU 측 최적화 + SSD 측 최적화

## 발표 자료 활용 포인트
- **HPCA Best Paper HM** — 최고 수준 학술적 인정

- 3계층 In-Storage 아키텍처 다이어그램
- Out-of-Order 샘플링 전후 비교: 순차 vs 재정렬
- BaM(GPU→SSD) vs BeaconGNN(SSD 내부) 비교 프레임

## 메모
- GNN의 극소 랜덤 I/O는 BaM [I1]도 겪는 문제 — BaM은 GPU 측 최적화, BeaconGNN은 SSD 측 최적화
- HPCA 2024에서 BeaconGNN + FlashGNN 동시 발표 → In-Storage GNN이 학술 트렌드
- OoO 샘플링의 아이디어는 CPU의 Out-of-Order 실행과 유사한 철학: 접근 순서를 재배열하여 하드웨어 병렬성 극대화
- 다이 레벨 부분 읽기는 기존 SSD가 하지 않는 혁신 → 하드웨어 수정 불가피
