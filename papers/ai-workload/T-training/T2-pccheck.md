# [T2] PCcheck: Persistent Concurrent Checkpointing for ML

- **학회/연도:** ASPLOS 2025
- **저자:** (Carnegie Mellon University 등)
- **분류:** 트레이닝 스토리지 최적화

## 핵심 요약 (1~2문장)
체크포인트를 트레이닝 연산과 완전히 병렬(concurrent)로 수행하는 persistent checkpointing 시스템. GPU 연산과 체크포인트 I/O가 동시에 진행되어 트레이닝 중단 시간을 사실상 제로에 가깝게 만든다.

## 읽기 전 질문
- "Persistent" concurrent checkpointing이란 기존 비동기 체크포인트와 무엇이 다른가?
- 체크포인트 일관성을 유지하면서 연산과 완전 병렬화하는 것이 어떻게 가능한가?
- 메모리 사용량과 대역폭 오버헤드는 어느 정도인가?

## 주요 내용 정리

### 1. 문제 정의 (Problem)
- 기존 비동기 체크포인트([T1] 등)도 GPU→CPU 복사 단계에서 일시적 중단 발생
- "완전히 중단 없는" 체크포인트가 이상적이지만, 모델 상태의 일관성 보장이 어려움
- 특히 분산 트레이닝에서 모든 노드의 상태가 동일한 step에 대응해야 함

### 2. 제안 방법 (Approach)
- **Persistent**: 체크포인트를 일시적 스냅샷이 아닌 지속적(persistent) 프로세스로 설계
- **Concurrent**: 트레이닝 연산과 체크포인트 I/O가 동시에 진행
- 모델 상태의 버전 관리를 통해 일관성 보장
- CUDA 스트림을 활용한 GPU→CPU 비동기 전송과 CPU→스토리지 파이프라인

### 3. 실험 결과
- ASPLOS 2025에서 발표된 최신 연구
- 체크포인트로 인한 트레이닝 throughput 저하를 사실상 제거
- 다양한 모델 크기와 분산 설정에서 검증

### 4. 한계점 및 향후 연구
- 추가 메모리 사용(스냅샷 버퍼)이 필요
- 매우 빠른 iteration 속도의 모델에서는 여전히 대역폭 한계 가능

## 다른 논문과의 관계
- 발전: [T1] DataStates-LLM의 비동기 접근을 더 발전시킨 형태
- 동기 부여: [W3] 85,000 체크포인트 (실제 필요성 실증)
- 비교: [T4] LMStor (다른 접근 — 계층적 체크포인트)

## 발표 자료 활용 포인트
- ASPLOS 2025 최신 연구로 학술적 가치 높음
- "체크포인트 오버헤드 제로"라는 목표의 실현 가능성 논의

## 메모
- 체크포인트 최적화의 끝판왕을 목표로 하는 논문
- 실제로 체크포인트 오버헤드가 제로가 되면, 체크포인트 빈도를 극대화하여 장애 복구 시간도 최소화 가능
