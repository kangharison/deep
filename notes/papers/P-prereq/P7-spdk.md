# [P7] SPDK: A Development Kit to Build High Performance Storage Applications

- **학회/연도:** CloudCom 2017
- **저자:** Yang et al.
- **분류:** 선행 연구 (Pre-2023)

## 핵심 요약 (1~2문장)
SPDK의 설계 철학과 구조: 유저스페이스 드라이버, 폴링 모드, 락프리 구조로 커널 오버헤드 제거. BaM이 GPU 측에서 유사한 접근을 구현.

## 읽기 전 질문
- 커널 NVMe 드라이버 대비 SPDK의 성능 차이는 어디에서 오는가?
- 폴링(polling) 모드가 인터럽트 방식보다 유리한 조건은 무엇인가?
- SPDK의 유저스페이스 드라이버가 다중 애플리케이션 환경에서 어떻게 동작하는가?

## 주요 내용 정리

### 1. 문제 정의 (Problem)
- NVMe SSD의 성능이 급격히 향상되어 커널 소프트웨어 스택이 오히려 병목
  - 커널 NVMe 드라이버: 시스템 콜 오버헤드, 인터럽트 처리 비용, 컨텍스트 스위칭
  - 블록 레이어 (bio/request 변환): 불필요한 추상화 오버헤드
  - 락(lock) 경합: 멀티코어 환경에서 I/O 큐 접근 시 동기화 비용
- 최신 NVMe SSD: 수백만 IOPS 가능하지만 커널 스택이 이를 소화하지 못함
- 레이턴시: 디바이스 자체 ~10μs인데 소프트웨어 오버헤드가 수십 μs 추가

### 2. 제안 방법 (Approach)
- **SPDK (Storage Performance Development Kit)**: 세 가지 핵심 원칙으로 커널 오버헤드 제거
  1. **유저스페이스 NVMe 드라이버**: 커널을 우회하여 유저스페이스에서 직접 NVMe 디바이스 제어
  2. **폴링(Polling) 모드**: 인터럽트 대신 완료 큐를 지속적으로 폴링하여 인터럽트 오버헤드 제거
  3. **락프리(Lockless) 설계**: 코어별 전용 큐 할당으로 락 없는 I/O 처리

```
기존 커널 스택:                    SPDK:
+------------------+              +------------------+
|   Application    |              |   Application    |
+--------+---------+              +--------+---------+
         |                                 |
+--------v---------+              +--------v---------+
|   System Call     |              |   SPDK Library   |
+--------+---------+              |   (유저스페이스)  |
         |                        |                  |
+--------v---------+              |  NVMe Driver     |
|   VFS / Block    |              |  (유저스페이스)  |
|   Layer          |              +--------+---------+
+--------+---------+                       |
         |                        +--------v---------+
+--------v---------+              |   NVMe Device    |
|   NVMe Driver    |              |   (MMIO 직접)    |
|   (커널)         |              +------------------+
+--------+---------+
         |                        시스템 콜 없음
+--------v---------+              인터럽트 없음
|   NVMe Device    |              락 없음
+------------------+
```

### 3. 핵심 아키텍처/설계
- **UIO/VFIO 기반 디바이스 바인딩**: 커널에서 NVMe 디바이스를 언바인드하고 유저스페이스에서 직접 접근
- **MMIO 직접 접근**: PCIe BAR를 유저스페이스에 매핑하여 NVMe 레지스터에 직접 읽기/쓰기
- **Submission/Completion Queue 직접 관리**: 유저스페이스에서 NVMe SQ/CQ를 직접 생성하고 doorbell을 직접 조작
- **코어별 전용 큐 페어**: 각 CPU 코어가 전용 SQ/CQ 페어를 가져 락 불필요
- **폴링 기반 완료 처리**: CQ를 바쁜 대기(busy-wait)로 폴링하여 완료 즉시 감지
- **비동기 I/O 모델**: 모든 I/O가 비동기로 제출되고 폴링으로 완료 확인
- **Blobstore/BlobFS**: SPDK 위에 구축된 경량 스토리지 추상화 계층

### 4. 실험 결과 (Key Results)
- **IOPS**: 커널 NVMe 드라이버 대비 **약 2~10x IOPS 향상** (특히 소규모 I/O에서 극적 차이)
- **레이턴시**: 평균 I/O 레이턴시 **약 50~70% 감소** (소프트웨어 오버헤드 제거)
- **CPU 효율**: 같은 IOPS를 달성하는 데 필요한 CPU 코어 수 대폭 감소
- 4KB 랜덤 읽기 기준: 단일 코어에서 **수십만 IOPS** 달성
- 다수의 NVMe 디바이스를 단일 코어에서 동시에 관리 가능
- 레이턴시 분포의 꼬리(tail latency)도 대폭 개선

### 5. 한계점 및 향후 연구
- **전용 코어 필요**: 폴링 모드로 인해 CPU 코어를 전용으로 할당해야 함 (유휴 시에도 100% 사용)
- **디바이스 독점**: 유저스페이스에 바인딩된 디바이스는 다른 프로세스/커널이 사용 불가
- **프로그래밍 복잡도**: 커널 파일시스템 대비 애플리케이션 코드가 복잡해짐
- **다중 테넌트 미지원**: 단일 애플리케이션이 디바이스를 독점하는 모델
- **파일시스템 부재**: BlobFS는 제한적이며 POSIX 호환 파일시스템 아님
- 향후: BaM [I1]이 GPU 측에서 유사한 유저스페이스 직접 접근 개념을 NVMe에 적용

## 다른 논문과의 관계
- 선행 연구: DPDK (네트워크에서의 유저스페이스 드라이버 개념)
- 후속 연구: BaM [I1] (GPU 측에서 유사한 유저스페이스/직접 접근 패러다임)
- 비교 대상: io_uring (커널 내 비동기 I/O 개선), libaio (리눅스 비동기 I/O)
