# [P4] EMOGI: Efficient Memory-access for Out-of-memory Graph-traversal in GPUs

- **학회/연도:** VLDB 2021
- **저자:** Min et al.
- **분류:** 선행 연구 (Pre-2023)

## 핵심 요약 (1~2문장)
GPU에서 호스트 메모리를 zero-copy로 직접 접근하여 대규모 그래프를 처리. PCIe 대역폭을 최대 활용하는 접근 방식으로 BaM의 직접적 선행 연구.

## 읽기 전 질문
- Zero-copy 방식으로 호스트 메모리에 접근할 때 PCIe 대역폭을 최대한 활용하는 비결은 무엇인가?
- GPU 스레드의 불규칙한(random) 메모리 접근 패턴이 PCIe 효율에 미치는 영향은?
- UVM(Unified Virtual Memory) 기반 접근과 zero-copy 직접 접근의 성능 차이는 얼마나 큰가?

## 주요 내용 정리

### 1. 문제 정의 (Problem)
- 대규모 그래프(수십~수백 GB)는 GPU 메모리에 들어가지 않음 (out-of-core)
- 기존 접근법:
  - **UVM (Unified Virtual Memory)**: CUDA의 관리형 메모리로 페이지 폴트 기반 데이터 마이그레이션 → 페이지 폴트 오버헤드가 매우 큼
  - **파티셔닝**: 그래프를 분할하여 반복적으로 GPU에 로드 → 파티셔닝 오버헤드와 중복 처리
- 그래프 탐색의 불규칙한 접근 패턴이 기존 방식의 성능을 심각하게 저하시킴

### 2. 제안 방법 (Approach)
- **Zero-copy 직접 접근**: GPU가 호스트 메모리(CPU DRAM)를 PCIe를 통해 직접 접근 (데이터 복사 없음)
- 핵심 인사이트: GPU의 수천 개 스레드가 동시에 PCIe 요청을 발생시키면, 개별 요청은 느려도 **집합적으로 PCIe 대역폭을 포화**시킬 수 있음
- 접근 패턴 최적화를 통해 PCIe 대역폭 활용률을 극대화

```
+------------------+     PCIe Gen3/4     +------------------+
|   GPU            | <=================> |   CPU Host       |
|   수천 스레드     |   zero-copy 접근    |   DRAM           |
|   동시 접근       |   (복사 없음)       |   [Graph Data]   |
|                  |                     |   수십~수백 GB    |
|   [GPU Mem]      |   대역폭 포화 전략  |                  |
|   (워킹셋만)     |                     |                  |
+------------------+                     +------------------+
```

### 3. 핵심 아키텍처/설계
- **Zero-copy 메모리 매핑**: `cudaHostRegister`로 호스트 메모리를 GPU 주소 공간에 매핑
- **Coalescing 최적화**: 같은 warp 내 스레드들이 인접한 호스트 메모리 주소를 접근하도록 데이터 레이아웃 재구성
- **워크로드 분류**:
  - 정점 데이터(degree 등): 작고 순차적 → GPU 메모리에 배치
  - 엣지 리스트: 크고 불규칙적 → 호스트 메모리에 유지, zero-copy 접근
- **그래프 데이터 구조 최적화**: CSR(Compressed Sparse Row) 포맷에서 엣지 리스트의 접근 효율성 향상
- **프리페칭 및 비동기 접근**: GPU 스레드 블록 수를 늘려 메모리 레이턴시를 숨김

### 4. 실험 결과 (Key Results)
- **UVM 대비 2~8x 성능 향상**: 페이지 폴트 오버헤드 제거로 대폭 개선
- **PCIe 대역폭 활용률**: PCIe Gen3 x16 기준 이론 대역폭의 **약 80~90%** 활용 달성
- BFS, SSSP, PageRank 등 다양한 그래프 알고리즘에서 일관된 성능 향상
- GPU 메모리에 완전히 들어가는 경우 대비 약 2~3x 느림 (PCIe 대역폭 한계)
- 최대 **128GB 이상**의 그래프 데이터를 GPU로 처리 가능

### 5. 한계점 및 향후 연구
- **PCIe 대역폭 한계**: zero-copy도 결국 PCIe 대역폭에 묶임 (GPU 메모리 접근 대비 10~20x 느림)
- **호스트 메모리 크기 제한**: 호스트 DRAM에 들어가지 않는 그래프는 처리 불가 → SSD 확장 필요 (BaM으로 발전)
- **CPU 메모리 간섭**: GPU의 대량 PCIe 접근이 CPU의 메모리 대역폭에 영향
- **그래프 전용 최적화**: 범용 워크로드에 대한 일반화는 추가 연구 필요
- 향후: BaM [I1]이 호스트 메모리를 넘어 SSD까지 zero-copy 확장, GIDS [I2]가 GNN에 특화

## 다른 논문과의 관계
- 선행 연구: CUDA UVM, GPUfs [P1]
- 후속 연구: BaM [I1] (호스트 메모리 → SSD로 확장), GIDS [I2] (GNN 특화)
- 비교 대상: CXL-BaM [I8] (CXL로 PCIe 대역폭 한계 극복)
