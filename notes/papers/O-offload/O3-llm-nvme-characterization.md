# [O3] An I/O Characterizing Study of Offloading LLM Models and KV Caches to NVMe SSD

- **학회/연도:** CHEOPS 2025
- **저자:**
- **분류:** GPU-NVMe 오프로드

## 핵심 요약 (1~2문장)
LLM 추론 시 모델 파라미터와 KV 캐시를 NVMe SSD로 오프로드할 때 발생하는 실제 I/O 패턴을 체계적으로 프로파일링한 특성 분석 연구. 오프로드 워크로드의 읽기/쓰기 비율, I/O 크기 분포, 순차/랜덤 패턴, 대역폭 활용률 등을 상세 측정하여 NVMe SSD 최적화를 위한 가이드라인 제시.

## 읽기 전 질문
- LLM 오프로드 워크로드의 I/O 패턴은 전통적인 스토리지 워크로드(DB, 파일서버 등)와 어떻게 다른가?
- 모델 파라미터 오프로드와 KV 캐시 오프로드의 I/O 특성 차이는 무엇인가?
- 현재 NVMe SSD 설계가 LLM 오프로드 워크로드에 최적화되어 있지 않은 부분은 어디인가?

## 주요 내용 정리

### 1. 문제 정의 (Problem)
- LLM 추론에서 GPU 메모리 부족을 해결하기 위해 NVMe SSD 오프로드가 활발히 연구됨
- 그러나 오프로드 워크로드의 **실제 I/O 특성**을 체계적으로 분석한 연구가 부족
- NVMe SSD는 전통적 워크로드(DB, 파일시스템)에 최적화되어 있어, LLM 오프로드 패턴에 맞지 않을 수 있음
- 오프로드 시스템 설계자들이 I/O 패턴에 대한 이해 없이 시스템을 구축하는 문제

### 2. 제안 방법 (Approach)
- **체계적 프로파일링 프레임워크** 구축
  - blktrace, ftrace, nvme-cli 등을 활용한 다계층 I/O 추적
  - DeepSpeed ZeRO-Inference, FlexGen, llama.cpp 등 주요 오프로드 프레임워크 대상
- **두 가지 오프로드 시나리오** 분석:
  - (1) 모델 파라미터(weights) 오프로드: 추론 시 레이어별 가중치를 SSD에서 로드
  - (2) KV 캐시 오프로드: long-context 추론 시 KV 캐시를 SSD에 저장/복원
- **다양한 모델 크기와 시퀀스 길이**에서 I/O 특성 변화 측정

### 3. 핵심 아키텍처/설계

```
  프로파일링 스택:
  ┌─────────────────────────────────────┐
  │     LLM 추론 프레임워크              │
  │  (DeepSpeed, FlexGen, llama.cpp)    │
  └──────────────┬──────────────────────┘
                 │  텐서 읽기/쓰기 요청
  ┌──────────────▼──────────────────────┐
  │     Python/C++ I/O Layer            │
  │  (mmap, read/write, GDS)           │
  └──────────────┬──────────────────────┘
                 │
  ┌──────────────▼──────────────────────┐
  │     Linux Block Layer (blk-mq)      │  ← blktrace 계측
  └──────────────┬──────────────────────┘
                 │
  ┌──────────────▼──────────────────────┐
  │     NVMe Driver                     │  ← I/O 크기, 큐 깊이 측정
  └──────────────┬──────────────────────┘
                 │
  ┌──────────────▼──────────────────────┐
  │     NVMe SSD                        │  ← SMART 데이터, 지연시간
  └─────────────────────────────────────┘

  관찰된 I/O 패턴:
  ────────────────────────────────────────
  모델 파라미터 오프로드:
  - 대부분 순차 읽기 (Sequential Read)
  - I/O 크기: 수 MB ~ 수백 MB (레이어 단위)
  - 읽기 위주 (Write 거의 없음)
  - 레이어별 주기적 패턴

  KV 캐시 오프로드:
  - 읽기/쓰기 혼합 (Read-Write Mix)
  - I/O 크기: 수 KB ~ 수 MB (토큰별)
  - 토큰 생성마다 쓰기, attention마다 읽기
  - 랜덤 접근 패턴이 더 빈번
```

### 4. 실험 결과 (Key Results)
- **모델 파라미터 오프로드**:
  - I/O가 거의 **100% 순차 읽기**로 구성
  - 평균 I/O 요청 크기 **수 MB~수십 MB** (NVMe에 유리)
  - NVMe SSD 대역폭 활용률: 최대 **80~90%** (순차 읽기 최적화 시)
  - 큐 깊이(Queue Depth)가 낮으면 대역폭 활용 저조 → 멀티큐 활용 필요
- **KV 캐시 오프로드**:
  - 읽기:쓰기 비율 약 **7:3 ~ 8:2**
  - I/O 크기가 **수 KB ~ 수백 KB**로 작아 SSD 효율 저하
  - 랜덤 I/O 비율 **40~60%** → SSD의 랜덤 읽기 성능이 중요
  - 시퀀스 길이가 길어질수록 KV 캐시 크기 급증
- SSD 종류(TLC vs QLC, PCIe Gen3 vs Gen4)에 따른 성능 차이 분석
- 오프로드 시 **NVMe 큐 깊이와 I/O 스케줄링**이 성능에 큰 영향

### 5. 한계점 및 향후 연구
- 학습(training) 워크로드가 아닌 추론(inference)에 초점 → 학습 시 I/O 특성은 다를 수 있음
- 제한된 수의 SSD 모델에서만 실험 → 더 다양한 SSD에서 검증 필요
- GDS(GPUDirect Storage) 경로의 상세 분석 부족
- NVMe SSD 내부 FTL의 동작(GC, WAF)과의 상관관계 심층 분석 부족
- 최적 SSD 설계 방향에 대한 구체적 제안이 제한적
