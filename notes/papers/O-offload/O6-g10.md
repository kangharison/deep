# [O6] G10: Enabling An Efficient Unified GPU Memory and Storage Architecture with Smart Tensor Migrations

- **학회/연도:** MICRO 2023
- **저자:**
- **분류:** GPU-NVMe 오프로드

## 핵심 요약 (1~2문장)
GPU 메모리와 NVMe SSD 스토리지를 통합 주소 공간으로 관리하는 하드웨어-소프트웨어 공동 설계 아키텍처. 텐서의 접근 패턴을 런타임에 분석하여 GPU VRAM과 SSD 사이에서 smart migration을 수행하며, DNN 학습/추론 워크로드에서 메모리 용량을 대폭 확장.

## 읽기 전 질문
- "Unified Memory and Storage Architecture"가 기존 UVM(Unified Virtual Memory)과 어떻게 다른가?
- Smart tensor migration의 "smart"한 부분은 구체적으로 무엇인가? (어떤 정책으로 마이그레이션 결정?)
- 하드웨어 수준 변경이 필요한가, 아니면 순수 소프트웨어 솔루션인가?

## 주요 내용 정리

### 1. 문제 정의 (Problem)
- DNN 모델 크기가 GPU 메모리 용량을 크게 초과 (예: GPT-3 175B → 350GB FP16)
- 기존 오프로드 방식들의 문제:
  - **수동 오프로드**: 프로그래머가 직접 데이터 이동을 관리 → 생산성 저하
  - **CUDA UVM(Unified Virtual Memory)**: page fault 기반으로 반응적(reactive) → 성능 저하
  - **정적 오프로드**: 컴파일 타임에 결정 → 동적 워크로드 변화에 대응 불가
- GPU 메모리 부족 시 OOM 에러 또는 배치 크기 제한으로 학습 효율 저하
- SSD를 단순 스왑 공간이 아닌, GPU 메모리의 일부로 통합하는 아키텍처 필요

### 2. 제안 방법 (Approach)
- **통합 주소 공간(Unified Address Space)**: GPU VRAM + NVMe SSD를 하나의 가상 메모리 공간으로 관리
- **Tensor-Level Migration**: 페이지 단위가 아닌 텐서 단위로 마이그레이션 수행
- **Proactive Migration**: 텐서 접근 패턴을 예측하여 사전에 마이그레이션 (reactive가 아닌 proactive)
- **Migration Policy**:
  - 텐서의 다음 접근 시점(next access time) 예측
  - GPU에 곧 필요한 텐서를 미리 SSD → GPU로 이동
  - 당분간 필요없는 텐서를 GPU → SSD로 이동
- **하드웨어 지원**: GPU MMU 확장을 통한 효율적인 주소 변환과 마이그레이션 트리거

### 4. 실험 결과 (Key Results)
- ResNet, VGG, BERT, GPT-2 등 다양한 DNN 모델에서 평가
- CUDA UVM 대비 **최대 3~5x 성능 향상** (proactive migration 효과)
- GPU 메모리를 SSD로 **4~16배 확장** 가능
- 이상적 상한(모든 데이터가 GPU에 있는 경우) 대비 **80~95%** 성능 달성
- OOM이 발생하던 대규모 배치에서 학습 가능
- 텐서 단위 마이그레이션이 페이지 단위 대비 **I/O 효율 2~3배 향상** (큰 연속 전송)
- SSD 대역폭 활용률 **70~85%** 달성

### 5. 한계점 및 향후 연구
- GPU MMU 확장 등 하드웨어 변경이 필요하여 기존 시스템에 즉시 적용 어려움
- DNN 이외의 불규칙한 접근 패턴 워크로드에서는 예측 정확도 저하
- PCIe 대역폭이 여전히 병목 → CXL 등 차세대 인터커넥트와의 통합 필요
- 시뮬레이션 기반 평가 위주 → 실제 시스템 구현 검증 부족
- 다중 GPU 환경에서의 확장성 검증 필요
